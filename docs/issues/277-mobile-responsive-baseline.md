## Narrative

The security-atlas web app is desktop-first. Tailwind v4 + shadcn/ui
give us responsive primitives for free, and ~10 of the ~20 authed
pages already use `md:`/`lg:` breakpoints (controls/[id], dashboard,
risks/hierarchy, calendar, settings, etc.) — but the foundational
plumbing for mobile is **missing**:

- **No `<meta name="viewport">` tag** in `web/app/layout.tsx`. Without
  it, mobile browsers render the page at desktop width (~980px) and
  zoom out, leaving operators with a microscopic UI even on pages
  whose Tailwind classes would have produced a perfectly usable
  mobile layout if the viewport were configured.
- **No mobile sidebar pattern**. `web/components/shell/sidebar.tsx`
  is permanently visible on every authed page. On a phone-width
  viewport it consumes ~40% of horizontal space and squeezes the
  content into an unusable strip.
- **No discipline doc** for "is this page mobile-ready?" — engineers
  adding new pages have no rubric to follow, and reviewers have no
  checklist.
- **No audit** of which existing pages already work at mobile widths
  vs which need follow-on attention.

This slice delivers the **foundational baseline** that makes the
existing responsive work actually visible to mobile users, plus a
documented audit + per-page tracker so the long tail of page-specific
optimizations can ship as targeted follow-on slices.

### What ships in this slice

1. **Viewport meta tag** in `web/app/layout.tsx` —
   `<meta name="viewport" content="width=device-width, initial-scale=1" />`
   via Next.js 16 App Router's `viewport` export pattern.
2. **Sidebar mobile-drawer behavior** — at `< md` breakpoint the
   sidebar collapses to a hamburger trigger that opens a slide-out
   drawer (shadcn `<Sheet>` component). At `≥ md` it renders inline
   exactly as today. Desktop UX unchanged (per the user's hard ask).
3. **`web/docs/responsive-discipline.md`** — single-page rubric:
   the Tailwind breakpoint contract (sm=640, md=768, lg=1024, xl=1280),
   the "test at 375px, 768px, 1280px" checklist, the rule "always
   start with `grid-cols-1` and grow with `md:grid-cols-N`", the
   ban-list for desktop-only patterns (e.g. hover-only affordances
   without `:focus-visible` parity), and the link to slice 056's
   responsive collapse for hierarchical-risk-style precedents.
4. **`docs/responsive-audit.md`** — per-page table of every authed
   route + chrome surface (sidebar/topbar/footer). Three columns:
   route, mobile-ready (yes/partial/no/n/a), notes/follow-on slice
   number. Generated by walking each route at 375px width and
   recording the failure mode (if any). Becomes the source of truth
   for the per-page spillover slices.
5. **Vitest coverage** for any new pure helpers (breakpoint-detection
   shouldn't be needed — Tailwind handles it CSS-side — but if any
   client-side viewport branching ships, it gets a pure helper +
   tests).
6. **Playwright coverage** — one new spec exercising sidebar drawer
   open/close at 375px viewport on `/dashboard` (the most-trafficked
   authed page). NOT a full per-page e2e fan-out; the per-page work
   ships in spillovers.

### Scope discipline (deliberately OUT)

- Per-page layout rework (e.g., turning the controls list table into
  a card stack at mobile). The audit doc identifies which pages need
  this; each lands as a separate spillover slice. The pattern is
  "Foundational baseline → audit → N small follow-on slices", **not**
  "everything in one mega-slice".
- Mobile-only routes (e.g., a separate `/m/dashboard`). The user
  picked "pure responsive CSS"; same routes, different layouts.
- Native iOS / Android apps. Out of scope. Forever, basically.
- Touch-vs-mouse interaction polish (e.g., long-press menus, swipe
  gestures). Defer to per-page slices where ergonomics warrant.

## Acceptance criteria

- [ ] **AC-1.** `web/app/layout.tsx` exports `viewport`
      (Next.js App Router pattern):
      `ts

  width: "device-width",
  initialScale: 1,

- [ ] **AC-2.** The rendered HTML on `/dashboard` includes
      `<meta name="viewport" content="width=device-width,initial-scale=1">`
      (curl-verifiable; asserted by a new vitest or playwright case).

### Sidebar drawer

- [ ] **AC-3.** At viewport width `< 768px` (Tailwind `md` breakpoint),
      the sidebar is hidden by default and a hamburger trigger button
      is visible in the topbar.
- [ ] **AC-4.** Clicking the hamburger opens a slide-out `<Sheet>`
      containing the same nav items as the desktop sidebar (including
      slice 213's audit pill + slice 214's count badges + slice 223's
      ⌘K + slice 250's avatar — composes with existing chrome).
- [ ] **AC-5.** At viewport width `≥ 768px`, the sidebar renders
      inline exactly as today. The hamburger trigger is hidden. **No
      desktop UX regression** (per the user's hard ask).
- [ ] **AC-6.** Drawer state is component-local (no persistence,
      no URL param). Clicking a nav item OR clicking outside the
      drawer closes it.
- [ ] **AC-7.** Drawer is keyboard-accessible (Escape closes; Tab
      cycles through nav items; first nav item receives focus on
      open).

### What goes in the discipline doc (AC-8)

The doc is the new-contributor onboarding artifact for "is this
page mobile-ready?". Sample structure:

```markdown
# Responsive discipline (security-atlas)

## Breakpoints (Tailwind v4 defaults)

- sm = 640px (small phone landscape / phablet portrait)
- md = 768px (tablet portrait / small laptop)
- lg = 1024px (tablet landscape / standard laptop)
- xl = 1280px (large laptop / desktop)
- 2xl = 1536px (large desktop)

## Test widths

Always test at 375 / 768 / 1280 before marking a page done.

## Default layout pattern

Start every multi-column layout at `grid-cols-1` and grow:
`grid grid-cols-1 gap-4 md:grid-cols-2 lg:grid-cols-3`

## Ban list

- Fixed pixel widths > 320 in containers
- Hover-only affordances without `:focus-visible` parity
- Horizontal scroll containers without a `<caption>` or
  ARIA equivalent
- Tables wider than 5 columns without an `md:` collapse pattern
  (slice 056's hierarchical-risk pattern is the canonical reference)

## Precedents

- Slice 056 (hierarchical risk dashboard) — responsive collapse
- Slice 256 (control-detail Coverage column) — mobile-aware table
- Slice 213 (audit pill) — drawer-compatible chrome
- Slice 223 (global ⌘K search) — drawer-friendly trigger
```
