# 070 — Onboarding walkthroughs (showboat-generated)

**Cluster:** Infra
**Estimate:** 2d
**Type:** JUDGMENT

## Narrative

Generate a small set of executable onboarding walkthrough documents that show — and prove, by re-executable code blocks — how the most load-bearing surfaces of security-atlas actually work. Aimed at newcomers (contributors, security engineers evaluating the project, auditors trying to understand what the platform does) who want to read the code paths, not just the architecture canvas.

**Terminology disambiguation (load-bearing):** this slice's "walkthrough" is the **PAI Walkthrough skill** (driven by `uvx showboat` — see `~/.claude/skills/Utilities/Walkthrough/walkthrough.md` for the canonical example). It produces a single markdown file that mixes prose with `bash`/code blocks whose captured output is committed alongside, so a reviewer can re-execute every block and confirm the document still matches reality. This is **distinct from** the audit walkthrough recording shipped in slice 027 (`internal/audit/walkthrough`), which is auditor-facing evidence capture against controls. The two concepts share a word and nothing else — every walkthrough doc shipped by this slice MUST live under `docs/walkthroughs/` (not `docs/audit-log/`, not `internal/audit/walkthrough/`), and the navigation header in each one MUST explicitly cite this distinction so future readers don't conflate them.

Five walkthroughs ship in this slice, each ~10 sections following the showboat canonical pattern (overview → ~8 traced code paths → "putting it all together" flow):

1. **`evaluation-pipeline.md`** — how an evidence record becomes a control state. Trace: connector pushes via Evidence SDK → schema-registry validates against `evidence_kind` → ingest writes append-only ledger row (slice 013/036) → eval engine computes per-cell state (slice 012) → control effectiveness rolls up over scope cells (slice 017/018) → API surfaces it at `/v1/controls/{id}/state`. Show real fixtures, real SQL, real OPA decisions.
2. **`audit-period-freezing.md`** — how an AuditPeriod transitions from `open` to `frozen` and what changes afterwards. Trace: period create (slice 028) → freeze sets `frozen_at` and hashes inputs (ADR-0003) → sample populations draw only from `observed_at ≤ frozen_at` (canvas §8.4) → live state continues independently → OSCAL export uses the frozen view (slice 030).
3. **`rls-tenant-isolation.md`** — how Postgres Row-Level Security actually enforces tenant isolation (constitutional invariant 6). Trace: `atlas_app` role (NOBYPASSRLS FORCE RLS) vs `atlas_migrate` role (BYPASSRLS) → tenancy middleware sets `app.current_tenant` (slice 033) → four-policy RLS pattern (`tenant_read`/`tenant_write`/`tenant_update`/`tenant_delete`) → live cross-tenant SELECT/INSERT/UPDATE/DELETE proving isolation against a fresh database from `bootstrap.sh`.
4. **`schema-registry-seed-and-validate.md`** — how `evidence_kind` schemas are seeded at boot and how validation rejects malformed records. Trace: `DefaultSeed()` registers platform kinds at startup (slice 014, with `.v1`-suffix canonical per `EVIDENCE_SDK.md` §4.5) → `LoadFromDB()` retry-with-backoff against the RLS app pool (slice 068 fix) → POST `/v1/evidence:push` invalidates a bad-payload then accepts a valid one.
5. **`oscal-ssp-export.md`** — how an SSP/AP/AR/POA&M bundle is generated for a frozen AuditPeriod (slice 030). Trace: Go calls the Python `oscal-bridge` over gRPC → compliance-trestle assembles the OSCAL document → ed25519 cosign-compatible signing → bundle is downloadable + verifiable.

Each walkthrough is generated by running `uvx showboat init`/`note`/`exec` commands against a hermetic local stack (docker-compose self-host bundle from slice 037, seeded by `fixtures/walkthroughs/`). The resulting markdown file ships in the repo verbatim — bash blocks AND captured output blocks, so a reviewer can replay them. A `just walkthroughs-refresh` recipe drives the regeneration end-to-end; it is on-demand only (no CI freshness gate — same anti-criterion as slice 057).

The set is integrated into the mkdocs Material site (slice 058) under a new "Walkthroughs" nav section so the docs site surfaces them to newcomers alongside the user-facing pages.

## Anti-criteria (P0 — block merge)

- **P0-A1**: Does NOT use the word "walkthrough" anywhere in the body without also citing the slice-027-vs-this disambiguation on first use in that file. Name collision with the audit-walkthrough concept is the dominant comprehension risk.
- **P0-A2**: Does NOT add a CI freshness gate. Walkthroughs are on-demand artifacts — same anti-criterion as slice 057's P0-A4 for screenshots. Drift signal is the showboat timestamp + reviewer discipline, not CI.
- **P0-A3**: Does NOT capture output that contains PII, real tenant names, real domain names, vendor token prefixes, the maintainer's name, or any secret material. Fixtures are neutral ("Acme Industries", "demo-tenant", `test-*` tokens) — same convention as slice 057's `fixtures/readme-demo/`.
- **P0-A4**: Does NOT manually hand-author the captured output blocks. Every output block in every walkthrough MUST be the actual `showboat exec` capture from a live local run. Hand-authoring output blocks defeats the showboat correctness guarantee that a reviewer can re-execute and confirm.
- **P0-A5**: Does NOT walkthrough surfaces that depend on un-merged features (forward-references are a slice 058 AC-10 mistake we are not repeating). If a walkthrough's premise depends on a slice not yet merged, the walkthrough does not ship in this PR — file a follow-up spillover slice instead.
- **P0-A6**: Does NOT duplicate the canvas. Walkthroughs trace **execution** through real code; canvas sections describe **intent + invariants**. Cite the canvas, do not paraphrase it.

## Notes for the implementing agent

- The single hardest decision call: **which five surfaces, in which order.** The five proposed in the narrative are an opinionated take — they cover the spine (eval), the auditor surface (freezing), the constitutional invariant (RLS), the evidence pipeline (schema registry), and the audit handoff (OSCAL). If the engineer's grill surfaces a sixth contender they think is more load-bearing for newcomers, swap one out and record the trade in the decisions log; do NOT ship six (scope creep).
- Showboat headers carry a UUID and a timestamp — both vary on every regeneration. Add them to a `.gitattributes` `merge=ours` rule if rebase conflicts become a problem, but most likely they won't (walkthroughs are not commonly co-edited).
- The five walkthroughs require a running stack with seeded data spanning evidence, controls, an open AuditPeriod, a freezable AuditPeriod, and a tenant that can demonstrate RLS isolation. `fixtures/walkthroughs/` should ship one fixture per walkthrough's data preconditions; the `walkthroughs-refresh` recipe applies them in order.
- Re-running showboat against the same stack should be deterministic ENOUGH that diffs on regeneration are small (one timestamp + one UUID per file). If a regeneration produces large diffs on output blocks, that's a real drift signal — the underlying surface changed, and the walkthrough needs review.
