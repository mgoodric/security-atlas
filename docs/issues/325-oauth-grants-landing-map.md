## Narrative

The 2026-05-27 architecture review (run via `voltagent-qa-sec:architect-reviewer` against `main` at `7f58715b`) flagged a contributor-onboarding gap in the OAuth Authorization Server substrate:

> "`internal/api/oauth/` — eight Go files plus integration tests for one wire surface. The proliferation reflects RFC compliance (RFC 6749, 7009, 7636, 7662, 8628, 8693, 9068) but the cognitive load is high. The package-level decomposition by grant type is reasonable. Future contributors will benefit from a one-page 'where does each grant land' map."

The substrate is correctly decomposed — each grant type / RFC concern lives in its own file (`authorize.go`, `token.go`, `revocation.go`, `oauthcode.go`, `user_resolver.go`, `password.go`, `oidc.go`, `sessions.go`, plus integration tests). ADR-0003 (`docs/adr/0003-oauth-authorization-server.md`) explains _why_ the substrate exists. What does not exist is a reference page mapping endpoints → grants → files → RFC sections so a new contributor can answer "which file handles the device-code grant?" or "where does PKCE validation live?" in one lookup.

This slice adds that reference map as a single contributor-facing docs page so the cognitive-load problem is closed once. The map is generated by reading the actual `internal/api/oauth/*.go` source — no aspirational rows — and cross-checked against `docs/openapi.yaml`.

**Scope discipline.** This slice adds one new reference doc, links it from ADR-0003, and updates the mkdocs nav if the doc lands in the docs-site. It does NOT:

- Restructure the `internal/api/oauth/` package layout. The decomposition is correct as-is.
- Add or remove any RFC-spec compliance.
- Touch any `.go` source file under `internal/api/oauth/` or `internal/auth/`.
- Substantively modify ADR-0003 (only adds a "See also" pointer).
- File the Evidence SDK docs alignment (slice 324) or the legacy-bearer responder retirement (slice 326).

- RFC 6749 (OAuth 2.0 Authorization Framework) — §4.1 (auth code), §4.3 (password), §6 (refresh)
- RFC 6750 (Bearer Token Usage)
- RFC 7009 (Token Revocation) — §2.1
- RFC 7636 (PKCE) — §4 (challenge), §4.6 (S256)
- RFC 7662 (Token Introspection) — §2.1
- RFC 8628 (Device Authorization Grant) — §3.1, §3.2, §3.4
- RFC 8693 (Token Exchange) — §2.1
- RFC 9068 (JWT Profile for Access Tokens) — §2

**Common-validator section.** Three to five cross-cutting helpers that ship across multiple grants. Read these from source before publishing:

- `internal/api/oauth/oauthclient/` — client authentication helpers
- `internal/api/oauth/pkce.go` — PKCE verification (if a separate file exists; otherwise inline in `authorize.go` / `token.go`)
- `internal/auth/tokensign/` — JWT minting / verification
- `internal/auth/keystore/` — signing-key management
- `internal/auth/jwtmw/middleware.go` — JWT-claim-extraction middleware
