# Slice 081 — Pre-push hook + slice-template step 9a guidance — decisions log

**Slice type:** `AFK` (the eight ACs are mechanically verifiable). This log records the build-time judgment calls the slice surfaced so post-deployment iteration is tractable. None of these blocked merge.

## Decisions made

### D1 — Hook delivery path: pre-commit-framework's `pre-push` stage, NOT a hand-rolled shell script

AC-1 says "populate `.husky/pre-push` (or equivalent path per repo's existing hook framework)." Reality check ruled out two alternatives:

1. **Husky / lefthook** — not present in the repo. `package.json` has no `husky` entry; no `.husky/` or `.githooks/` directory exists; `git config core.hooksPath` is unset. Installing one would violate **P0-A2** ("Does NOT install the hook system if one isn't already there").
2. **Hand-rolling `.git/hooks/pre-push` as a committed file** — `.git/hooks/` is not version-controlled, so the script would have to live somewhere in the repo and a separate installer would copy/symlink it. Plus the existing `.git/hooks/pre-push` is owned by **agentdiff** (header: `agentdiff pre-push hook — managed by agentdiff init / DO NOT EDIT — regenerate with: agentdiff init`). Overwriting it would break agentdiff trace push.

Resolved: wire pre-commit-framework's native `pre-push` stage. The same hooks defined in `.pre-commit-config.yaml` run on whichever stages the user installs (`pre-commit install -t <stage>`). `just install-hooks` now runs `pre-commit install --hook-type pre-push --install-hooks` in addition to the existing `pre-commit install --install-hooks`.

Verified upstream behavior (pre-commit 4.6.0 — `pre_commit/commands/install_uninstall.py`, lines 76–86): when `pre-commit install` encounters an existing non-pre-commit hook at the target path, it `shutil.move`s the existing hook to `<hook-type>.legacy` and writes its own hook script. The pre-commit-framework hook then invokes `_run_legacy` (`pre_commit/commands/hook_impl.py` line 269) which executes the `.legacy` script with the same args. agentdiff's `pre-push` becomes `pre-push.legacy` and continues to run on every push, transparently.

**Confidence: high.** The migration behavior is a documented design choice of pre-commit-framework and is verified against upstream source.

### D2 — Hook-file path actually used: `.git/hooks/pre-push` (managed by pre-commit-framework, chains to `.git/hooks/pre-push.legacy`)

Per D1. There is no new file committed for the hook itself — the hook is generated by `pre-commit install -t pre-push`, which is now part of `just install-hooks`. The agentdiff hook is preserved as `.git/hooks/pre-push.legacy` and continues to run.

**Confidence: high.** Verified by reading pre-commit-framework source.

### D4 — `--no-verify` bypass: preserved and documented, NOT prohibited

Per **P0-A1** the hook must allow `git push --no-verify`. pre-commit-framework's pre-push hook is invoked by `git push` via the standard git hook mechanism; `git push --no-verify` skips ALL pre-push hooks by git's own logic (no work on our part required to preserve this).

CONTRIBUTING.md "Local CI parity" subsection documents the bypass and warns against casual use, citing the recurring pre-commit-failure data ("5 of the 62 CI failures observed on 2026-05-15"). The warning is one sentence — long enough to be honest about why the hook exists, short enough to not read like a manifesto.

**Confidence: high.** Git semantics + observed pre-commit-framework behavior both confirm.

### D5 — AC-5 deliberate-failure test design

The hook must demonstrably block a push when pre-commit would catch something. Picked **trailing-whitespace** as the deliberate-breakable change: it is one of the `.pre-commit-config.yaml`-installed hooks (`trailing-whitespace` from `pre-commit-hooks` v4.6.0), it is deterministic (no auto-fix race), and it is easy to introduce/revert.

Test procedure + result:

1. Locally unset `core.hooksPath` (this worktree had `core.hooksPath` set by agentdiff init — pre-commit refuses to install when that's set; `git config --unset core.hooksPath`).
2. `pre-commit install --hook-type pre-push --install-hooks` — output: `Running in migration mode with existing hooks at .../pre-push.legacy` (agentdiff hook preserved as `.legacy`) + `pre-commit installed at .../pre-push`. **Migration verified empirically.**
3. Created `docs/AC5_DELIBERATE_TEST.md` with trailing whitespace (3 spaces before `\n` on line 3, verified via `od -c`).
4. `git commit --no-verify -s -m "test: AC-5 deliberate prettier-breakable fixture"` — commit succeeded (--no-verify skipped pre-commit so the commit could land with the bad content).
5. `git push --dry-run origin HEAD:refs/test/081-prepush-deliberate` — output:
   ```
   trim trailing whitespace.................................................Failed
   - hook id: trailing-whitespace
   - exit code: 1
   - files were modified by this hook

   Fixing docs/AC5_DELIBERATE_TEST.md
   ...
   error: failed to push some refs to 'https://github.com/mgoodric/security-atlas.git'
   ```
   **Hook blocked the push as designed.** Error message is clear, names the offending hook, names the offending file, points the engineer at the fix.
6. `--no-verify` bypass verified: `git push --no-verify --dry-run origin HEAD:refs/test/081-prepush-bypass` skipped the hook entirely and reached the network layer (`* [new reference]   HEAD -> refs/test/081-prepush-bypass`). **`--no-verify` works exactly as required by P0-A1.**
7. Cleanup: `git reset --soft HEAD~1` + delete the test fixture + restore `core.hooksPath`. Working tree clean of test artifacts.

**Confidence: high.** Both blocking and bypass paths verified empirically.

### D6 — Beyond status-flip: the `_STATUS.md` post-rebase re-pad pattern is in the same class

The narrative footnote mentions "`_STATUS.md` re-pad after merge-queue rebases — slightly different shape from the status-flip pattern but same fix." Slice 081's pre-push hook also catches this case: after a rebase, the engineer's local tree has the rebased `_STATUS.md` content; `git push` invokes pre-commit on the diff `origin/main..HEAD`; if prettier would reformat anything, the push is blocked. No additional work needed in this slice — it falls out of the same mechanism.

**Confidence: medium.** Confirmed in principle; not exercised in AC-5's deliberate test. Re-evaluate when the next merge-queue rebase happens against a `_STATUS.md`-touching slice.
