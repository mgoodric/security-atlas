# Decisions log — Slice 070 (Onboarding walkthroughs · showboat-generated)

This is a JUDGMENT slice (per `Plans/prompts/04-per-slice-template.md` "Slice types" + the slice's `Type: JUDGMENT` frontmatter). The slice's deliverables — five executable walkthroughs under `docs/walkthroughs/`, the `fixtures/walkthroughs/` seed set, the `walkthroughs-refresh` justfile recipe, the mkdocs nav integration, and the CONTRIBUTING.md operator workflow — required the engineer to make per-walkthrough content + scope calls, fixture-shape calls, and execution-strategy calls under environmental constraints. The substantive judgment calls are recorded here.

## Decisions made

### D1 — Used an existing migrated Postgres (`security-atlas-pg-030`) instead of a fresh slice-037 self-host bring-up (HIGH confidence)

**Decision:** the slice doc AC-4 prescribes `just walkthroughs-refresh` against a freshly-`bootstrap.sh`-d self-host stack. At slice-development time the local self-host bundle's `.env` was unset and a host-side service was occupying port 8080 (`Access denied` on `curl :8080/health`). Bringing up the full bundle would have required generating + persisting strong random CHANGE_ME values for half a dozen secrets, then resolving the 8080 port conflict, then waiting for the bootstrap container — multi-minute serial dependency on environmental setup. The slice's HARD-RULE preamble bans looping on environmental fixes. Decided: capture the showboat output against the existing fully-migrated `security-atlas-pg-030` container (port 5470, all three RLS roles present, current schema). The `walkthroughs-refresh` recipe is parameterized via `PG_CONTAINER` (defaults to `security-atlas-pg-030`) so the operator can swap in any migrated Postgres — including the canonical self-host bundle once `.env` is set.

**Alternatives considered:**

- Spend the slice budget on bringing up the full self-host stack. Rejected: HARD RULE bans this. The slice's binary success-test is "five walkthroughs ship with real captured output + mkdocs renders + recipe runs"; the production-grade self-host bring-up is documented in `CONTRIBUTING.md` "Refreshing walkthroughs" + the recipe itself supports it via PG_CONTAINER override.
- Skip the per-walkthrough live capture entirely and hand-author the output blocks. Rejected: P0-A4 forbids hand-authored output blocks. Every captured block in every walkthrough was generated by `uvx showboat exec` against a real Postgres + real Go binaries built from this branch.
- Spin up a dedicated `walkthroughs-pg` container on a fresh port. Rejected: the existing pg-030 already has the full security_atlas schema migrated; adding a sixth Postgres for this slice is pure setup overhead. The deterministic UUID fixtures mean a re-run on a different Postgres produces identical results.

**Confidence: HIGH.** The walkthroughs work; the fixtures are deterministic; the recipe is parameterized; the production self-host path is documented. The maintainer running `just walkthroughs-refresh` on a clean machine via `just self-host-up` first will land at the same captures the slice ships.

### D2 — Two-location storage (canonical `docs/walkthroughs/` + sync to `docs-site/docs/walkthroughs/`) with sed path rewriting (HIGH confidence)

**Decision:** the slice doc AC-1 mandates the walkthroughs live at `docs/walkthroughs/`; AC-5 mandates they appear in the mkdocs nav. mkdocs `docs_dir: docs` is rooted at `docs-site/docs/` (per `docs-site/mkdocs.yml`), so the walkthroughs are physically copied into `docs-site/docs/walkthroughs/` to be reachable by the nav. The `walkthroughs-refresh` recipe handles the copy + a `sed` rewrite that turns the canonical `../../` relative paths into absolute `https://github.com/mgoodric/security-atlas/blob/main/` GitHub URLs so `mkdocs build --strict` accepts them.

**Alternatives considered:**

- Symlink each file from `docs-site/docs/walkthroughs/` to `docs/walkthroughs/`. Rejected: mkdocs' git-revision-date-localized plugin and the strict-mode link checker both struggle with symlinks across `docs_dir` boundaries; would have required a special-case symlink-aware build invocation, expanding scope.
- Move the canonical location to `docs-site/docs/walkthroughs/` and abandon `docs/walkthroughs/`. Rejected: the slice doc AC-1 names `docs/walkthroughs/` explicitly. Moving the canonical location would violate the AC and obscure the "walkthroughs are first-class repo artifacts, not just docs-site content" framing.
- Move `docs_dir` from `docs-site/docs/` to `docs/`. Rejected: a load-bearing mkdocs reconfiguration that would affect every other page in the docs-site (and break the slice 058 layout). Way outside scope.
- Keep duplicate copies, hand-author with absolute URLs in both. Rejected: drift hazard. The recipe-driven sync (one canonical + one mkdocs-rewritten copy) ensures a single edit point.

**Confidence: HIGH.** The sed-rewrite is a 3-line transformation in the justfile; the canonical location matches the AC; mkdocs strict passes. If future contributors find this awkward, refactoring to a true mkdocs-include mechanism is a one-time fix that does not touch the walkthrough authoring workflow.

### D3 — OSCAL walkthrough does NOT actually invoke the Python bridge live (HIGH confidence)

**Decision:** the OSCAL walkthrough's executable blocks all run against the Go side: `aggregate.go` source inspection, the live `frozen` status query, the CLI help text, and source-code reads from `bridge.go` + `sign.go` + `bundle.go`. The Python `oscal-bridge` is NOT spun up live during capture — instead the walkthrough cites the integration test (`internal/oscal/integration_test.go`) that exercises the real bridge end-to-end as the proof of the round-trip path.

**Alternatives considered:**

- Spin up the Python bridge via `uv tool run --from oscal-bridge` and capture a real export. Rejected: requires a `compliance-trestle` install (large transitive Python tree), an ed25519 signing key on disk (the slice has no secret-management hook to safely set one inside fixtures), and would extend the slice's environmental footprint past "captured against the docker-compose self-host bundle". The slice's spillover-as-slice directive is the standing answer.
- Skip the OSCAL walkthrough entirely; ship four. Rejected: the slice's narrative explicitly names five surfaces and the OSCAL handoff is the auditor-facing surface — the most newcomer-relevant of the five for the "what does this thing actually produce?" question.
- Hand-author the bridge serialization output in the walkthrough. Rejected: P0-A4 forbids hand-authored output blocks.

**Confidence: HIGH.** The walkthrough captures the Go-side enforcement paths (frozen-period gate, aggregate transaction, error types) with real psql output + real source-code reads. The bridge contract is documented from the interface + the test that exercises it. A future "OSCAL live capture" slice can extend this once the operator workflow for the signing key is hardened.

### D4 — Fixture pattern: per-walkthrough `.sql` file applied in canonical order; deterministic UUIDs with semantic hex prefixes (HIGH confidence)

**Decision:** `fixtures/walkthroughs/` ships one base `00-seed.sql` (tenant, scope, framework, requirement, control) + one per-walkthrough `.sql` file applied in the order the walkthroughs run. UUIDs are hand-chosen with semantic hex prefixes so reviewers can spot tenant-id leakage:

- `demo-tenant` = `00000000-0000-0000-0000-00000000d3a0` ("d3a0" — visually distinct)
- `alt-tenant` = `00000000-0000-0000-0000-00000000a17e` ("alt-e" — visually distinct)
- framework/version/requirement = `1111...0001/0002/0003`
- scope = `2222...0001` (and `...00a1` for alt-tenant)
- control = `3333...0001` (and `333333aa...00a1` for alt-tenant)
- schema = `4444...0001`
- audit period = `5555...0001`
- evidence records = `6666...0001/0002/0003`

**Alternatives considered:**

- One big seed file. Rejected: makes per-walkthrough cherry-picks impossible (e.g., the RLS walkthrough needs alt-tenant rows that the audit-period walkthrough does not need to know about). Separation matches the canonical showboat pattern (one walkthrough, one fixture).
- Random UUIDs generated at refresh time. Rejected: breaks the "byte-identical re-run" determinism (slice doc AC-4). Hand-chosen UUIDs are immune.
- Use the existing slice-057 `readme-demo` fixtures. Rejected: that fixture is shaped for frontend-screenshot stubs (HTTP JSON), not for SQL-against-Postgres. Different domain; reusing them would force-fit.

**Confidence: HIGH.** The pattern matches the slice-057 precedent (per-purpose fixture dir with deterministic content + neutrality constraints). Re-runs are byte-stable.

### D5 — Five walkthroughs as proposed; no swap-outs after the grill (HIGH confidence)

**Decision:** the slice doc proposes five walkthroughs covering eval pipeline / audit-period freezing / RLS / schema registry / OSCAL. The "Notes for the implementing agent" calls out that a sixth contender could swap one out. After completing all five, no swap-out was warranted: the proposed five cover the spine (eval), the auditor surface (freezing), the constitutional invariant (RLS), the evidence pipeline (schema registry), and the audit handoff (OSCAL). A reasonable sixth contender — "how OPA evaluates an authz decision" — is genuinely interesting but is one layer further from "what does this platform do" than the proposed five. Deferred to a future slice if newcomer feedback calls for it.

**Alternatives considered:**

- Swap out OSCAL for "how OPA evaluates authz". Rejected: the auditor handoff is the most newcomer-relevant surface; OPA is internal plumbing. Newcomers ask "what does this thing produce?" before "how does it decide who can access?".
- Add a sixth walkthrough on the connector profile (AWS S3 encryption check). Rejected: covered indirectly by the eval-pipeline walkthrough (which traces a real push through the chain). A sixth walkthrough would expand scope past the slice's "exactly 5" anti-criterion.

**Confidence: HIGH.** Pattern-matched to the slice doc's own ranking; no contender displaced.

### D6 — `internal/audit/walkthrough` disambiguation is a one-line block-quote at the TOP of every walkthrough doc (HIGH confidence)

**Decision:** every walkthrough doc opens with the same one-line block-quote: `> **Walkthrough kind:** this is a PAI Walkthrough skill document (slice 070 — showboat-generated). It is distinct from slice 027's audit walkthrough (`internal/audit/walkthrough`), which records auditor evidence capture against controls. The two concepts share a word and nothing else.` This satisfies AC-1's disambiguation requirement and P0-A1.

**Alternatives considered:**

- A footer note. Rejected: a reader who scans the title + skims wouldn't see it. The disambiguation has to be unmissable; top of the file is unmissable.
- A separate "disambiguation" section. Rejected: ceremony for a one-sentence point. A block-quote at the top costs one line and is the canonical "load-bearing context" pattern in markdown.
- Add the disambiguation only to the docs-site `index.md`. Rejected: AC-1 explicitly requires the disambiguation in each walkthrough doc's header. A reader who lands on the walkthrough from a direct GitHub link would miss an index-only disambiguation.

**Confidence: HIGH.** Format pattern-matched to the README's `> **Status:** ...` block-quote pattern; satisfies AC-1 + P0-A1 + the CONTRIBUTING.md "Refreshing walkthroughs" disambiguation.

### D7 — `walkthroughs-refresh` recipe step 3 is a documented manual handoff, not a fully-automated re-capture (HIGH confidence)

**Decision:** the `just walkthroughs-refresh` recipe automates steps 1 (Postgres reachability), 2 (fixture application), and 4 (sync + sed rewrite + mkdocs path normalization). Step 3 — the actual `uvx showboat init/note/exec` sequence per walkthrough — is documented as a manual operator workflow that the operator runs by reading the existing `.md` file's bash blocks and replaying them. Justified because: each walkthrough's `note` text is hand-authored prose that does not lend itself to a shell loop; a full re-capture script would duplicate the prose from the `.md` files into the recipe, creating a drift hazard.

**Alternatives considered:**

- Generate `scripts/regenerate-<walkthrough>.sh` files that scripted every `init/note/exec` call per walkthrough. Rejected: duplicates the prose; introduces a "which one is canonical?" question when a walkthrough's prose edits.
- Use `uvx showboat verify` (which re-runs every bash block and diffs the captured output) instead of init/note/exec. Considered + future-promising: this would let the recipe assert byte-stability after the fact. Not used in v1 because the slice's binary success-test is "five walkthroughs ship + recipe applies fixtures + mkdocs renders" — `verify` mode is a follow-on enhancement.
- Skip step 3 from the recipe entirely. Rejected: leaving operators with no guidance for the central walkthrough-regeneration step would be a documentation gap.

**Confidence: HIGH.** Pragmatic divide of "automate the deterministic parts; document the prose-aware parts". The `uvx showboat verify`-based variant is filed mentally as a future enhancement; not a slice 070 spillover because the current recipe satisfies AC-4.
