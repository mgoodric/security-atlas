# Slice 339 — decisions log

**Slice:** 339 — OpenAPI spec drift: OAuth endpoints not enumerated in `docs/openapi.yaml`
**Type:** JUDGMENT
**Date:** 2026-06-14

The OAuth Authorization Server surface (slices 187-192) was registered in
`internal/api/oauth/oauth.go` but absent from the canonical REST spec
`docs/openapi.yaml`. This slice closes that drift. The build-time judgment
calls below were made by the implementing agent and are recorded here rather
than blocked on a human sign-off (per the JUDGMENT slice convention).

---

## D1 — Fidelity reinterpretation: generated-spec, not hand-authored RFC schemas (Option A)

**Decision.** Add the OAuth endpoints to `internal/api/openapi/RouteSpecs` so
they appear in the generated `docs/openapi.yaml` at **the same fidelity as
every other route** (path / method / tag / tier / summary). The slice doc's
AC-2 / AC-4 / AC-7 (rich per-RFC request/response/error schemas + shared
component `$ref`s) are reinterpreted to "consistent-with-every-other-endpoint
generated fidelity."

**Why.** `docs/openapi.yaml` is **generated**, not hand-edited. The header of
the file says `# AUTOGENERATED by cmd/atlas-openapi … DO NOT EDIT BY HAND`,
and the BLOCKING `openapi-drift-check` CI job asserts the committed file is
byte-identical to the generator output. The generator
(`internal/api/openapi/generator.go`) consumes only the `RouteSpec` struct
fields — `Method`, `Path`, `Tag`, `Tier`, `Internal`, `Summary` — and emits a
single `default` response that `$ref`s the shared `Ack` envelope for **every**
operation. There is no per-route schema field. Hand-adding rich schemas to the
generated file is impossible: the next drift-check regen would diff and fail.
All ~270 pre-existing routes are at this minimal fidelity; documenting the
OAuth endpoints at a higher fidelity than the rest of the spec is not
expressible and would make the spec internally inconsistent.

**Rejected: Option B (populate generator-supported schema fields).** The
generator supports no per-route schema fields, so there is nothing to
populate. Not viable.

**Rejected: extend the `RouteSpec` struct + generator to carry full RFC
schemas.** That is a materially larger piece of work (a request/response/error
schema model, component extraction, validator changes) — out of scope for a
0.5-1d docs-drift slice, and it would touch the generated shape of all 280
routes. Captured as a spillover candidate (see "Spillover" below); not
implemented here.

---

## D2 — Coverage-scanner extension was MANDATORY, not optional

**Decision.** Extend `scripts/check-openapi-drift.sh` with a third extraction
block ("block C") that resolves the oauth package's const-based route
registrations to their literal paths.

**Why it was required (not a scope creep).** The drift-check has two halves:
(1) inventory drift (committed spec == generator output) and (2) coverage
drift, which scans `internal/api/*/` for chi route registrations and asserts
each is declared in `RouteSpecs`. The coverage scan only matched string
literals on the `/v1/`, `/auth/`, `/health` prefixes (plus a
`root.Method(http.MethodX, "literal")` form for `/metrics`). The OAuth routes
are registered as `r.Get(PathJWKS, …)` / `r.Post(PathToken, …)` — a **const
identifier, not a string literal**, on the `/oauth/` and `/.well-known/`
prefixes that the scan never matched. That is exactly why these routes lived
in code without tripping the BLOCKING check.

The consequence: adding the routes to `RouteSpecs` puts them in the scanner's
"declared" set but NOT its "actual" set, which trips the **stale-entry** half
of the coverage check (`declared-but-not-registered`) and fails CI. So the
scanner extension is not optional — without it the slice's own gating AC
(drift-check green) cannot pass. This is the inconsistency the slice brief
flagged: "adding the routes to RouteSpecs without scanner coverage creates an
inconsistency you must resolve."

**Shape of the fix.** Block C builds a const→literal map from the oauth
package's `PathX = "/literal"` definitions, then maps each `r.METHOD(PathX, …)`
Mount call to `METHOD literal-path`, feeding those into the same "actual" set
the existing two blocks populate. Verified in isolation: it resolves to exactly
the 9 registered routes. Bonus: future oauth route additions are now guarded by
the same BLOCKING check (the gap that let this drift exist is closed).

**Portability note.** macOS (BSD) `sed` supports neither `\b` nor `\s`; an
initial draft using them silently no-op'd block C on the local repro while
still working on GNU sed in CI. The committed patterns use `[ ]*` and explicit
character classes so block C behaves identically on GNU sed (CI/Linux) and BSD
sed (macOS/local). This matches the project's "run every CI tool locally"
discipline — a local-vs-CI sed delta would otherwise hide the bug.

**Not touched:** two pre-existing SC2086 shellcheck infos on the script's
error-printing `printf` lines (178/188) are outside this diff and left as-is
(scope discipline).

## D3 — Code-vs-doc endpoint delta: 9 registered, not the slice doc's 12

**Decision.** Document only the **9** routes the chi router actually registers.
Code is ground truth (anti-criterion P0-339-2: "spec must match shipped
behavior").

**The delta.** The slice doc (and the brief) listed 12 endpoints. Three of
them do **not** exist in `main`:

| Listed in slice doc                       | In code? | Disposition                                                                |
| ----------------------------------------- | -------- | -------------------------------------------------------------------------- |
| `POST /oauth/authorize` (consent)         | No       | Only `GET /oauth/authorize` is registered; there is no POST consent route. |
| `POST /oauth/device_authorization/verify` | No       | No `PathDeviceVerify` const, no route, no handler anywhere in the repo.    |
| `POST /oauth/userinfo`                    | No       | No `userinfo` reference anywhere in `internal/`, `cmd/`.                   |

Confirmed three ways: the `Mount` method in `oauth.go`, a repo-wide grep for
`userinfo`/`verify`/`PathDeviceVerify`, and the existing
`internal/api/testdata/routes.golden` (a `chi.Walk` capture of the assembled
router) — all agree on exactly these 9:

1. `GET  /.well-known/jwks.json`
2. `GET  /.well-known/openid-configuration`
3. `GET  /oauth/authorize`
4. `POST /oauth/token`
5. `POST /oauth/revoke`
6. `POST /oauth/introspect`
7. `POST /oauth/device_authorization`
8. `POST /oauth/device_authorization/approve`
9. `POST /oauth/device_authorization/deny`

Documenting the 3 phantom endpoints would also have violated P0-339-2 directly
(aspirational endpoints not in `main`). They are not filed as spillovers
because there is no evidence they are planned; if `POST /oauth/authorize`
(consent), device verify, or userinfo land in a future slice, that slice owns
adding its own RouteSpec entry (now guarded by block C).

## D4 — Tag and tier choices

**Tag = `oauth`** for all 9. The brief allowed `oauth` or `OAuth AS`; `oauth`
matches the existing lowercase, single-word tag convention in `RouteSpecs`
(`auth`, `controls`, `risks`, `ucf`, `oscal`, …). AC-6 (grouped in generated
docs) is satisfied — they all share one tag and render together.

**Tier = `none`** for all 9. Rationale:

- The `RouteSpec.Tier` field maps only to the three platform security schemes:
  `none` (no `security:` block), `bearer`, `adminBearer` — i.e. the platform
  **JWT** the AS itself mints.
- Per `internal/api/register_platform.go`, the entire oauth mount is bearer-
  and authz-**exempt**: these endpoints ARE the auth system; they authenticate
  OAuth **clients** (and end-users at the authorize/device steps), not platform
  users holding an atlas JWT.
- `/oauth/revoke` and `/oauth/introspect` do require **client** authentication
  per RFC 7009 / RFC 7662 (RFC 6749 §2.3.1), but that is `client_secret_basic`
  / `client_secret_post` handled **inside** the handler (e.g. `revoke.go` reads
  the `Authorization` header), NOT the platform `bearer`/`adminBearer` JWT
  middleware. The `RouteSpec.Tier` enum cannot express "client_secret auth", so
  forcing `bearer` here would be wrong (it would advertise a platform-JWT
  requirement that does not exist and is not enforced by middleware).
- This is the same precedent as the `/auth/*` sign-in routes, which are all
  `Tier: "none"` for the identical reason (credential-in-body, no platform JWT
  yet). The summaries note "client auth in body/header" for revoke/introspect
  so the in-handler requirement is still surfaced to a spec reader.

## Detection-tier classification (slice 353, Q-13)

- `detection_tier_actual`: `manual_review` — the drift itself was found by a
  human/agent cross-reference during slice 325 (D7), not by an automated tier.
  The root cause (the coverage scanner's blind spot for const-based,
  non-standard-prefix registrations) was likewise found by reading the scanner
  during this slice, not by a failing check.
- `detection_tier_target`: `integration` (the BLOCKING `openapi-drift-check`
  CI job is the project's spec-coverage tier). This slice **moves the actual
  toward the target**: block C makes the coverage scanner catch any future
  `/oauth/*` or `/.well-known/*` route that is registered but not declared, so
  the next instance of this class of drift fails CI rather than waiting for a
  manual cross-reference.

## Spillover

- **Candidate (NOT filed): generator rich-schema support.** Extending
  `RouteSpec` + the generator to carry per-route request/response/error schemas
  (the literal reading of AC-2/4/7) is a real enhancement but a much larger,
  cross-cutting slice that would re-shape all 280 routes. It is deferred as a
  reinterpretation here (D1) rather than filed, because it is a spec-wide
  fidelity decision the maintainer should scope deliberately, not a follow-on
  obligation created by this slice. If the maintainer wants it, it should be
  filed as its own slice with its own design (schema model, component
  extraction, validator changes). No new slice doc was created to avoid filing
  an under-specified placeholder.

No spillover slice files were created.
