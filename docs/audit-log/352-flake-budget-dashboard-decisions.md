## D1 — Counter cadence: weekly cron, not on-merge

**Decision:** The flake counter runs on a weekly cron (`0 9 * * 1` —
Mondays 09:00 UTC, matching the existing `edge-image-prune.yml`
cadence) plus `workflow_dispatch` for ad-hoc invocations. Not on
every merge to `main`.

**Rationale:**

- The budget AC-3 floor is "at least weekly". On-merge cadence would
  exceed the floor (~5-20 merges/day during high-velocity batches)
  for no commensurate signal value — the dashboard is read by humans
  on a maintenance-review rhythm, not a per-merge rhythm.
- On-merge would multiply API call cost by ~50x. The 90-day baseline
  walk already consumes ~13 chunked API calls + per-attempt-jobs
  calls for the rare multi-attempt SHAs (~2 minutes wall-clock).
  Doing that on every merge is wasteful.
- Concurrency on the dashboard commit becomes a real concern with
  on-merge — two parallel merges produce two counter runs that both
  try to push `docs/flake-budget-dashboard.md` to `main`. The
  weekly singleton avoids the race entirely.
- Matches the project's existing weekly-cron pattern (`edge-image-prune.yml`,
  slice 207), so the maintainer's cron-review surface stays at one slot.
- Alternative: GitHub Actions workflow trigger `workflow_run` listening
  for ci.yml completion. Rejected — too noisy (every PR / merge fires
  it), and the dashboard would commit-storm.

**Revisit trigger:** if the maintainer finds the weekly cadence too lossy
(flake events expire from a 7-day window before being noticed), step
up to twice-weekly. If the on-merge information is genuinely needed,
add a second workflow that emits a slack ping but does NOT touch the
dashboard.

**Confidence:** HIGH. The weekly cadence is the cheapest correct shape;
on-merge would be premature optimization in the wrong direction.

## D2 — "Flake" definition: re-run-cleared-on-same-SHA only

**Decision:** v1 counts a flake only when a CI job for a given
`(head_sha, surface)` failed on attempt N and succeeded on attempt
M > N, same SHA, no code change in between. The definition does NOT
include the weaker signal of "failure on commit A, success on commit
A+1 without test-code change".

**Rationale:**

- The strict signal is unambiguous from the GitHub Actions REST API:
  one query per multi-attempt SHA yields the exact answer. No
  judgment, no false positives.
- The weaker signal (fail-on-A, pass-on-A+1) conflates flake with
  fix-forward. Without diffing test code between A and A+1, we can't
  tell which it was. A diff-based heuristic would compound the
  counter's complexity for marginal signal.
- The slice doc explicitly anticipates v1 simplicity ("shell script
  that greps gh run list --json output for test-failure patterns is
  fine for v1").
- The strict signal is what most professional flake-tracking tools
  (Datadog CI Test Visibility, BuildPulse, Trunk Flaky Tests) call
  "true flake".

**Alternatives considered:**

1. **Weaker signal (fail-on-A, pass-on-A+1).** Rejected for v1 —
   needs a diff-based exclusion of "the fix landed in A+1" and the
   complexity isn't worth it before we know how often the strict
   signal fires.
2. **Test-level flake detection (parse logs for `--- FAIL: TestX`).**
   Deferred to v2. Requires fetching every job log (expensive — gigabytes
   per week), parsing test names, and per-test attribution. The
   dashboard's "top-3 flaking tests" column is therefore "—" for v1;
   the appendix lists the failing-run URLs instead, which is enough
   information for a human triage.
3. **Counter that fires only on `main`-branch runs.** Rejected —
   most flakes are caught on PRs before merge; restricting to main
   under-counts.

**Revisit trigger:** if v1 reports zero flakes for two consecutive
months (the 90-day baseline already shows zero — see D4 below) and
the maintainer's anecdotal sense is that flakes are happening, widen
the definition in a v2 slice to include the fail-on-A / pass-on-A+1
signal. The widening is a budget-document edit (versioned).

**Confidence:** HIGH for v1 scope; calibration-in-progress for v2.

## D3 — Cheap denominator: count attempts from run-level conclusion

**Decision:** The "attempts per surface" denominator is computed
cheaply: for every workflow run in the window with a non-skipped,
non-cancelled overall conclusion, count +1 attempt for ALL four
surfaces. We do NOT make a per-job API call for every single-attempt
SHA.

**Rationale:**

- Per-job API calls for the typical case (single-attempt, single-success
  SHA) would multiply API cost by ~50x (each run has ~50 jobs).
- The approximation overcounts when path-filtering stubs out a
  specific surface job — but the path-filter pattern is uniform across
  the four surfaces (a docs-only PR stubs out ALL four equally), so
  the bias is symmetric and rates remain comparable.
- The exact case — per-job attempt counting — only matters when
  detection-phase activity already pulled the per-job data (multi-
  attempt mixed-conclusion SHAs); those cases get exact treatment.
- The dashboard's methodology block documents this approximation
  explicitly so future maintainers know the bias.

**Alternative considered:** fetch per-job data for every run. Rejected
on cost — 1000 runs \* 50 jobs = 50k API calls per 7-day window. The
GitHub Actions REST API rate limit is 5000/hr; we'd burn the budget
on weekly cron alone.

**Revisit trigger:** if a future change to the path-filter logic
introduces an asymmetric skip pattern (e.g. only the Playwright
surface is path-filtered, the others always run), revisit. Add a
per-surface skip-rate row to the dashboard so the bias is observable.

**Confidence:** MEDIUM-HIGH. The bias is small and symmetric today;
it could grow if path-filter logic evolves.

## D4 — Baseline finding: zero flakes in 90 days

**Decision:** Document the 90-day baseline result openly in both the
budget doc and the dashboard, even though the result ("zero flakes")
might appear to undermine the slice's premise.

**Result:** 90-day baseline (2026-02-27 → 2026-05-28):

| Surface         | Flakes | Attempts | Rate  | Target | Status |
| --------------- | ------ | -------- | ----- | ------ | ------ |
| Go unit         | 0      | 2003     | 0.00% | 0.0%   | green  |
| Go integration  | 0      | 2003     | 0.00% | 0.5%   | green  |
| Frontend vitest | 0      | 2003     | 0.00% | 0.0%   | green  |
| Playwright e2e  | 0      | 2003     | 0.00% | 1.0%   | green  |

**Rationale for surfacing the finding:**

- The result is honest signal about the v1 definition's narrowness.
  Slice 340's chromedp flake (5 consecutive failures across slices
  312 / 315 / 320) DOES exist in CI history, but each occurred on a
  different SHA — under the strict v1 definition those are not "flakes",
  they are "real failures the maintainer eventually re-investigated".
- The baseline is the right calibration point for "is this budget too
  aggressive or too lax?" In our case it's "the strict definition is
  silent on the project's actual recent flake history" — which is
  important to know because it tells future maintainers the trigger
  rate is structurally low.
- The strict definition is NOT useless: it WILL catch a real flake
  the moment someone hits "Re-run all failed jobs" on a PR and the
  re-run passes. That's a clean signal. The 0/90d means nobody has
  needed to use re-run during the window — or the maintainer has been
  fix-forwarding instead.

**Implication for the budget:** the per-surface targets (0% / <0.5% /
~0% / <1%) are well within the observed signal's noise floor. The
budget's value is forward-looking — when the project lands a flake-
prone area (new chromedp surface, new postgres-RLS suite, new
playwright spec exercising a race-prone modal), the counter will catch
it. The 90-day baseline is the floor, not the ceiling.

**Revisit trigger:** if the v1 definition reports zero flakes for
TWO MORE quarters (Q2 + Q3 2026), widen the definition in a v2 slice.

**Confidence:** HIGH on the data; MEDIUM on whether 0/90d means "v1 def
too narrow" or "project is actually flake-clean."

## D5 — Trigger automation: file issue, do NOT auto-quarantine

**Decision:** When a surface crosses its trigger threshold (e.g.
2 Go-integration flakes in 7 days), the counter files a GitHub issue
labeled `flake-investigation` with the offending test/run details.
The counter does NOT:

- `t.Skip` the failing test
- Add `test.skip` to a Playwright spec
- Modify any product code
- Auto-close the PR that observed the flake
- Re-run the failing job

The maintainer triages: file a slice (slice 340 pattern), or close
the issue as resolved if the flake was already fixed-forward.

**Rationale:**

- P0-2 explicitly forbids auto-quarantine. The counter is a signal,
  not an actor. Auto-quarantine inverts the merge-block discipline
  (slice 322) — the budget is for AGGREGATE rate tracking, not
  per-incident allow-listing.
- Filing an issue is the lightest-weight handoff to a human. The
  maintainer can decide in 30 seconds whether to act.
- Idempotency: if an open `flake-investigation` issue for the same
  surface already exists in the trigger window, the counter appends
  a comment rather than filing a duplicate. Prevents weekly-cron
  issue spam.

**Alternative considered:** Slack notification instead of GH issue.
Rejected — the project's primary collaboration surface is GitHub
issues; a slack ping disappears from history.

**Confidence:** HIGH. The shape directly matches slice 333's
recommendation and P0-2.

## D6 — Workflow commits dashboard back to main directly

**Decision:** The `flake-counter.yml` workflow commits the regenerated
`docs/flake-budget-dashboard.md` directly to `main` using the workflow
GITHUB_TOKEN. Not via PR.

**Rationale:**

- The dashboard is a derived artifact. The budget document
  (`docs/flake-budget.md`) is the source of truth; the dashboard is
  computed output. Computed output via PR is friction for no review
  value — there's nothing to approve, the script ran.
- Matches the existing pattern in the project for derived artifacts
  committed by automation (e.g. dependabot lockfile updates land via
  PR because the lock files DO need review; the dashboard does not).
- Branch protection on `main` does NOT block GITHUB_TOKEN pushes by
  default (verified via `.github/branch-protection.json`). The
  workflow has explicit `contents: write` permission for this purpose.
- A bad dashboard commit is revertable like any other commit. The
  blast radius is one markdown file.

**Alternative considered:** open a PR for each refresh. Rejected — adds
PR noise to the queue, and merge-friction reverses for no review value.

**Revisit trigger:** if branch protection ever tightens to block
GITHUB_TOKEN pushes, the workflow fails loudly and the maintainer
notices. At that point, switch to PR-based.

**Confidence:** HIGH.

## D7 — Chunked API calls to work around the 1000-result cap

**Decision:** The script slices the requested window into 7-day chunks
and walks each chunk independently, then deduplicates by run id. This
works around the GitHub Actions `workflow_runs` REST endpoint's
1000-result cap.

**Rationale:**

- A high-velocity 7-day window already produces ~1000 runs (980 in
  the 7-day baseline). A 90-day window would silently truncate to 1000
  without chunking — we'd miss 90% of the data.
- The chunk size of 7 days is empirical: 1054 runs in the 7-day
  window suggests we're at the cap boundary. A finer chunk (e.g. 3
  days) would be more cautious but doubles the API-call overhead.
- The dedupe step handles chunk-boundary overlap defensively (a run
  whose `created_at` is within a second of the boundary may appear
  in both chunks).

**Alternative considered:** filter by branch + paginate (the per-branch
endpoint has a different cap shape). Rejected — multiplies complexity
without solving the underlying cap problem for `main`-branch
high-velocity periods.

**Revisit trigger:** if the project's CI run volume grows past 2000/week,
the 7-day chunk also hits the cap. At that point, drop chunk size to
3 days (still cheap).

**Confidence:** HIGH. Validated against the actual 90-day baseline
(2466 runs, all surfaced).

## D8 — Workflow runs prettier after the script for idempotence

**Decision:** The `flake-counter.yml` workflow invokes
`npx --yes prettier@3.3.3 --write docs/flake-budget-dashboard.md`
after the script writes the dashboard, before the commit step.

**Rationale:**

- The script writes compact markdown tables (`| col | col |`); prettier
  rewrites them to column-aligned tables (`| col   | col   |`). Without
  the normalization step, every cron run would produce a "prettier
  reformat" diff on top of the script's output, and the workflow's
  `git diff --quiet` check would never see "no change" — it would
  commit a near-empty diff weekly.
- Replicating prettier's column-alignment in pure bash is fragile
  (requires per-table column-width pass with unicode-width awareness
  for the `·` character in CI job names).
- `npx` per-invocation is ~3s on `ubuntu-latest` (node is preinstalled
  on the runner image). Acceptable for weekly cadence.
- Prettier version pinned to `3.3.3` — matches the project's
  `.pre-commit-config.yaml` `mirrors-prettier@v3.1.0` resolution
  (verified at slice-land time; revisit if pre-commit pinning drifts).

**Alternative considered:** make the script emit prettier-compatible
output directly. Rejected — column-alignment requires unicode-width-
aware multi-pass formatting; the npx step is cheaper.

**Confidence:** HIGH.
