# Walkthroughs

Executable, source-of-truth onboarding documents that show — and prove,
by re-executable code blocks — how the most load-bearing surfaces of
security-atlas actually work. Aimed at newcomers (contributors, security
engineers evaluating the project, auditors trying to understand what the
platform does) who want to read the code paths, not just the
architecture canvas.

## Distinct from slice 027

The walkthroughs in this section are PAI Walkthrough skill documents
(slice 070, generated by `uvx showboat`). They are **unrelated** to
[`internal/audit/walkthrough`](https://github.com/mgoodric/security-atlas/tree/main/internal/audit/walkthrough)
(slice 027), which records an auditor's evidence capture against
controls. The two concepts share a word and nothing else. Every
walkthrough doc's header restates this disambiguation so future readers
don't conflate them.

## How to read one

Each walkthrough is a single markdown document with three parts:

1. An **overview** linking back to the constitutional invariant /
   canvas section / slice doc(s) the walkthrough demonstrates.
2. **~8 traced sections**, each mixing prose with one fenced `bash`
   block and the captured output of that block (every output frame is
   real — generated by `uvx showboat exec` against a live local stack
   seeded by `fixtures/walkthroughs/`).
3. A final **Putting it all together** section that summarizes the
   chain, plus a **Where to read more** pointer to (a) the canvas
   section, (b) the slice doc(s), and (c) the Go package path.

Walkthroughs are reproducible: a reviewer with the same fixtures and a
running stack can re-execute every block via
`just walkthroughs-refresh` and confirm the captured output still
matches reality (see CONTRIBUTING.md "Refreshing walkthroughs").

## When to add a new walkthrough

Walkthroughs are deliberately scarce. Each one is a real maintenance
surface: when the underlying code changes, the walkthrough must be
re-captured (`just walkthroughs-refresh`) and its diff reviewed. A
sixth walkthrough is justified only when:

- the surface is constitutional-invariant-load-bearing (or a P0
  anti-criterion enforcement point), AND
- the existing architecture canvas section does not, by itself, give a
  reader enough to trace the code path, AND
- a hands-on traversal of the surface materially helps newcomers.

If those three are true, propose it in a new slice doc with the
`Type: JUDGMENT` frontmatter (per
`Plans/prompts/04-per-slice-template.md`). Slice 070's anti-criterion
P0-A5 explicitly bars forward-referencing un-merged features.
