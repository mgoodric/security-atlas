# OAuth grants landing map

> Reference. "Which Go file handles grant X?" answered in one lookup.

The OAuth Authorization Server substrate lives under `internal/api/oauth/` and the supporting primitives under `internal/auth/`. ADR-0003 (`docs/adr/0003-oauth-authorization-server.md`) records _why_ the substrate exists; this page records _where each grant lands_ in the source tree.

The map is sourced by reading `internal/api/oauth/*.go` handler registrations directly — not by inferring from the ADR. Endpoints that are documented elsewhere but not yet registered in `main` are intentionally omitted (no aspirational rows).

## How to read this page

- The first table maps every registered OAuth endpoint and grant type to the file that owns its handler.
- The "Common validators" table lists cross-cutting helpers the per-grant files all call into.
- The "Where the JWT claims come from" subsection points at the canonical claim shape and explains the tenant-switch contract.
- The "Endpoints not in the OpenAPI spec" callout records a known gap (filed as a separate spillover slice).

### Endpoints not in the OpenAPI spec

The endpoints listed above are wired in Go but are not currently enumerated in `docs/openapi.yaml`. The discrepancy is tracked separately (spillover from slice 325 — file as a small follow-on slice if not already filed). The OpenAPI spec MUST eventually match the wire reality; this map is the interim source of truth.

### Grants intentionally omitted

| Grant                                                                       | Why omitted                                                                                                                                                                                                                                |
| --------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `grant_type=password` (RFC 6749 §4.3 — Resource Owner Password Credentials) | Not implemented. First-party login goes through the OIDC RP and the `/oauth/authorize` + authorization-code + PKCE path. The `internal/auth/password` package hashes human credentials with argon2id but is not exposed as an OAuth grant. |
| `grant_type=refresh_token` (RFC 6749 §6)                                    | Not implemented in v1. Clients re-acquire after the 1-hour access-token lifetime expires. Deferred per ADR-0003 / P0-188-6.                                                                                                                |
| `response_type=token` (RFC 6749 §4.2 — Implicit grant)                      | Rejected at `/oauth/authorize` (P0-189-4). Industry best practice is to use authorization-code + PKCE in its place.                                                                                                                        |

## Where the JWT claims come from

The canonical claim shape is `jwt.AtlasClaims` in `internal/auth/jwt/claims.go`. The atlas-namespaced claims locked at canvas OQ #21 Reading D (2026-05-20):

| Claim                     | Go type                  | Meaning                                                                                                                                                                                          |
| ------------------------- | ------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `atlas:idp_issuer`        | `string`                 | The upstream OIDC issuer that authenticated the human. Pinned at code-redemption time; carried unchanged across tenant switches.                                                                 |
| `atlas:current_tenant_id` | `uuid.UUID`              | The tenant the bearer is currently scoped to. THIS is the only claim that changes on tenant switch — and only via the RFC 8693 token-exchange grant.                                             |
| `atlas:available_tenants` | `[]uuid.UUID`            | The set the user is a member of, captured at code-redemption from the `users` table via the BYPASSRLS pool (slice 192).                                                                          |
| `atlas:roles`             | `map[uuid.UUID][]string` | RBAC roles per tenant. Keyed by tenant UUID, value is the role list.                                                                                                                             |
| `atlas:super_admin`       | `bool`                   | Platform-wide super-admin flag. Membership lives in the no-RLS `super_admins` table per slice 198; the claim is _copied_ from the verified subject and can never be synthesized true (P0-188-4). |

Validation runs in this fixed order in `jwt.Validate`: `iss` → `aud` → `exp` → `nbf` → tenant-scope (`current_tenant_id ∈ available_tenants`). Each check has a named sentinel error (`ErrIssuerMismatch`, `ErrAudienceMismatch`, `ErrExpired`, `ErrNotYetValid`, `ErrTenantOutOfScope`) so the JWT middleware can discriminate.

### Tenant switching never mutates an in-flight claim

`atlas:current_tenant_id` is the only claim that changes when the operator switches tenants in the UI. The change happens by minting a NEW JWT via the RFC 8693 token-exchange grant against `/oauth/token` — never by editing an existing token. The old JWT continues to verify until its `exp` (or until `/oauth/revoke` is called against it); the new JWT carries the new `current_tenant_id` and the same `available_tenants[]`. This is the eventual-eviction contract documented in `docs-site/docs/tenant-membership.md`.
